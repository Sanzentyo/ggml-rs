### 🚨 なぜ現在の設計のままではダメなのか（課題点）

1. **コンパイル時の型の喪失（データ型の不一致）**
`Tensor` 構造体が「何のデータ型か（f32かi32か）」を知らないため、「f32のテンソルにi32を書き込む」といったバグがコンパイル時に弾けません。
2. **コンパイル時の形状の喪失（次元の不一致）**
テンソルが形状情報を持たないため、行列積（`mul_mat`）などで次元が噛み合わないテンソル同士を渡しても、実行してエラーになるまで気づけません。
3. **引数スワップの温床（生の `usize` の多用）**
行（Rows）と列（Cols）の引数に `usize` を直接使っているため、利用者が順番を逆に渡してもコンパイラは警告してくれません。
4. **APIの肥大化とサフィックス地獄（C言語的設計）**
関数名に型や引数の種類がハードコードされています。その結果、本来1つで済むはずのメモリ計算関数が `recommended_matmul_memory_f32_shapes_bytes` のように6つにも分裂・乱立しており、将来の拡張（f16対応など）を著しく阻害しています。

### ✨ どうすべきか（完全版・Rustらしい解決策）

1. **要素型のジェネリクス化（`<T>` と `GgmlElement` トレイト）**
テンソルを `Tensor<'ctx, T>` にし、型ごとのメソッド（`set_f32` など）をジェネリックな `write_data(&[T])` に一元化して、型の不一致をコンパイルエラーにします。
2. **冗長なサフィックスの廃止とAPIの統合**
ジェネリクスを活用し、乱立していた関数群を統合します。例えば、6つあったメモリ計算関数は `recommended_matmul_memory::<T>(lhs: Shape2D, rhs: Shape2D) -> Result<Bytes>` という1つの関数にまとめます。
3. **強い型（`shape` モジュール）の全面適用**
作成された `Cols`、`Rows`、`Shape2D`、`Bytes` をすべてのAPIに強制適用します。関数のシグネチャ自体をドキュメント化し、引数の渡し間違いを完全に排除します。
4. **Const Genericsを用いた形状の静的検証（ハイブリッド設計）**
`StaticShape2D` を活用し、コンパイル時に行列積の次元一致などを証明できる `StaticTensor<'ctx, T, S>` を導入します。ただし、推論時の動的なコンテキスト長（LLMの特性）にも対応できるよう、基本は動的テンソル（`Tensor`）とし、形状が確定している箇所のみ静的テンソルにダウンキャストできる「ハイブリッド設計」を採用します。
